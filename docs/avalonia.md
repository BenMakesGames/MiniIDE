# Avalonia gotchas

- `TabControl.DataTemplates` (Control-inherited collection) selects the content template by the item's runtime type. Use it instead of `ContentTemplate` when items are heterogeneous (e.g. editor tab + image tab under one `TabControl`). Each `DataTemplate` needs an explicit `DataType`. `ItemTemplate` (tab header) can target the common base type because it only binds base members.
- `Avalonia.Data.Converters.ObjectConverters.IsNotNull` (and `IsNull`) are the idiomatic converters for gating `IsVisible` on a reference property being null/non-null. Wire with `Converter={x:Static ObjectConverters.IsNotNull}`. No custom `IValueConverter` needed.
- `Avalonia.Media.Imaging.Bitmap` ctor decodes synchronously and throws on failure. Wrap in `try/catch` when decoding user-supplied image paths; store the message for display.
- `IClassicDesktopStyleApplicationLifetime.Args` holds the launch arguments (nullable `string[]?`), already stripped of the exe path — `Args[0]` is the first real argument, do not skip it. Populated from the `args` passed to `StartWithClassicDesktopLifetime`. Guard for null before indexing/enumerating.
- A `ContextMenu` (and its `MenuItem`s) inherits the `DataContext` of the control it's attached to. On a control with no bound item — e.g. a top-bar `TextBlock` under the window — that resolves to the *window* `DataContext` (the VM), not a per-item model. Any `Click` handler reading `(sender as MenuItem)?.DataContext` must handle that VM type, or it silently no-ops. Shared path-resolver `switch`es should include a VM arm alongside the item-type arms.
- `ContextMenu.Opening` is a `System.ComponentModel.CancelEventHandler` (`(object? sender, CancelEventArgs e)`) — **synchronous**, so per-item enablement computed here cannot `await`; use only cheap cached state. Resolve the individual `MenuItem`s from `sender`: `((ContextMenu)sender).Items.OfType<MenuItem>()` matched on `mi.Name`. Named items **inside a `DataTemplate`** aren't in the window's namescope, so `this.FindControl<MenuItem>("...")` won't find them — resolve from the menu instead.
- To spawn a **visible** external terminal (vs. piping output), set `ProcessStartInfo.UseShellExecute = true` — the opposite of the output-capturing services (`RunService` uses `false`). `wt.exe` (Windows Terminal) is an app-execution alias that may be absent; a failed `Process.Start` throws `Win32Exception`. Attempt `wt.exe` (`ArgumentList` `-d <dir> <cmd>`), and on throw set an instance flag and fall through to `powershell.exe` (`WorkingDirectory = <dir>`, `ArgumentList` `-NoExit -Command <cmd>`). The flag makes subsequent invocations skip `wt.exe` for the app's lifetime. Only a *launch* failure of `wt.exe` should trip the fallback — a command failing inside a successfully-spawned terminal is not observable here and must not.
- `MenuItem.InputGesture` is **display-only** — it renders the gesture text in the item's right-aligned column and registers *no* key handler. `MenuItem.HotKey` is what actually binds the key. Use `InputGesture` to advertise a shortcut that's already handled elsewhere (e.g. a global `KeyDown` handler) with zero double-fire risk. Accepts a `KeyGesture` string parsed at load (`"F12"`, `"Shift+F12"`, `"Ctrl+Shift+F"`), and coexists with a dynamically-set `Header` since it's a separate column.
- `DoubleTapped` only fires when **both** pointer presses resolve to the *same* source element ([Avalonia #8733](https://github.com/AvaloniaUI/Avalonia/issues/8733)). On templated item rows (e.g. `TreeViewItem`) where the content doesn't fill the container, the two clicks can straddle content vs. container padding → no `DoubleTapped`. Use a `PointerPressed` handler acting on `e.ClickCount == 2` instead — it fires wherever the press registers (everywhere selection already works). Register it **tunneling** (`AddHandler(PointerPressedEvent, h, RoutingStrategies.Tunnel)`) so it runs before the item consumes the press for selection, and do **not** set `e.Handled` (selection must still commit on that press). `ClickCount` is not source-sensitive — confirmed reliable across full row incl. edges/right-of-text.
- `TreeViewItem` has **no** `HorizontalContentAlignment`/`VerticalContentAlignment` (it derives from `HeaderedItemsControl`, not `ContentControl`) — a `Setter` for them fails to compile. Its Fluent template template-binds the header `ContentPresenter`'s `HorizontalAlignment`/`VerticalAlignment` to the *item's own* `HorizontalAlignment`/`VerticalAlignment`, and the default theme sets `VerticalAlignment="Center"` — so the presenter is shorter than the row, leaving top/bottom dead strips. To make row content fill the full height (uniform hit target), set `VerticalAlignment="Stretch"` on the item via a `TreeView.ItemContainerTheme` → `ControlTheme TargetType="TreeViewItem" BasedOn="{StaticResource {x:Type TreeViewItem}}"`. Horizontal is already `Stretch` and `Padding` already `0` in the Fluent theme. `BasedOn` the default (don't replace the template) or the chevron/indent/selection break.
- A `Button` can carry **both** `Command="{Binding …}"` and a code-behind `Click="…"` handler — both fire on a click, and **neither** fires while the button is disabled (`CanExecute` false). Use this to bolt a view-only side effect (e.g. surfacing a tab) onto a command-bound button without a `CanExecute` re-check in the handler and without pushing view state into the VM — the disabled state gates the click for free.
- Select a `TabItem` programmatically from code-behind by resolving both the `TabControl` and the target `TabItem` by `x:Name` (`this.FindControl<TabControl>("…")` / `FindControl<TabItem>("…")`) and setting `tabs.SelectedItem = tab`. Keep tab-activation a view concern here — don't model "selected tab" in the VM.
- **Heterogeneous `TreeView` nodes** (e.g. group headers vs. leaves): put a `TreeDataTemplate DataType="…" ItemsSource="{Binding Children}"` for parents **and** a plain `DataTemplate DataType="…"` for leaves in `TreeView.DataTemplates` (not `ItemTemplate`). Avalonia resolves per node by runtime type — same mechanism as `TabControl.DataTemplates`. A leaf `DataTemplate` with no children just renders a row.
- A control inside a **non-selected `TabItem`** (hand-authored `TabItem`, not `ItemsSource`) is not in the visual tree at window-ctor time, so `AddHandler(...)` in the ctor can miss it. Hook it via `AttachedToVisualTree="…"` on the control + a `bool` once-guard (mirrors the editor-binding pattern) — fires when the tab is first realized.
- **One realized control is shared across all same-`DataType` items in a `TabControl`.** When you switch between two tabs whose VMs resolve to the *same* `TabControl.DataTemplates` entry, Avalonia does **not** realize a fresh control per tab — it reuses the one control and swaps its `DataContext`. So any per-instance wiring done in code-behind (imperative `TextEditor.Document` assignment, event-handler subscriptions) must **rebind on every `DataContextChanged`**, not once on `AttachedToVisualTree`: keep a `Dictionary<Control, Binding>` of per-control state, guard with `ReferenceEquals(state.CurrentVm, newVm)` to skip redundant rebinds, and **detach the previous item's handlers** before wiring the new one — else handlers leak and stale documents keep firing. (A one-shot `HashSet<Control>` "already bound" guard is the trap: it binds the first item and silently no-ops for the rest.) Switching to a *different* `DataType` realizes a different control (different template), so that path is fine.
- **`DataContext == ActiveTab` is ambiguous once two tab kinds render the same control type.** Code that finds "the active editor" via `descendants.OfType<TextEditor>().FirstOrDefault(e => e.DataContext == ActiveTab)` breaks the moment another tab kind (e.g. an output tab) also hosts a `TextEditor` — an active output tab now matches. Constrain the predicate by VM type too: `e.DataContext is EditorTabViewModel && e.DataContext == ActiveTab`.
- **Binding an enum to `RadioButton`s** without a converter: expose one `bool` facade per value (`get => Enum == X; set { if (value) Enum = X; }`), bind each radio's `IsChecked` two-way to a facade, and in the enum's `partial void On…Changed` call `OnPropertyChanged` for every facade so the others uncheck. Clean for a 2–3 value toggle; reach for a converter only past that.
- A `[RelayCommand]`'s `CanExecute` that depends on **non-observable** service state (e.g. `SolutionService.SolutionPath`) won't re-evaluate when that state changes — expose a `NotifyCanRefreshChanged()` that calls `SomeCommand.NotifyCanExecuteChanged()` and have the owner invoke it after the state transition. `[NotifyCanExecuteChangedFor(nameof(SomeCommand))]` on an `[ObservableProperty]` covers the observable half automatically.
- A `MenuItem.Icon` whose `TextBlock` has an **explicit `Foreground`** keeps full color when the item is disabled — the hard-coded brush overrides the theme's disabled-foreground, so a greyed item shows a bright icon. Dim the whole item with a scoped `MenuItem:disabled → Opacity` style (put it in `ContextMenu.Styles`; `0.4` matches the `ComboBoxItem:disabled` precedent) — one item-level opacity dims icon + text together, no per-child binding. Same hazard bit the `ComboBoxItem` icon templates.
- Once **any** `MenuItem` in a menu carries an `Icon`, Avalonia reserves the icon gutter for **every** sibling — iconless items render blank-but-indented, aligned under the icons. Expected, not a regression.
- Icon glyphs on **static** `MenuItem`s (whose `DataContext` is the VM or a per-item model that doesn't carry the glyph) come via `{x:Static m:SomeIconClass.Xxx}`, not `{Binding}` — the static reference is `DataContext`-independent, so the same value works across menus attached to different sources. Needs the `xmlns:m="using:…Models"` declaration (already on the window root here).
- Don't `await` startup work inside `OnFrameworkInitializationCompleted` — it runs before the window is shown and stalls startup. Construct the VM into a local, bind it to the window, then `Dispatcher.UIThread.Post(async () => await vm.SomeCommand.ExecuteAsync(...))` to defer the work until after the window is up. Reuse the same VM instance the window is bound to — don't construct a second one.
